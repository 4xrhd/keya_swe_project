# Software Requirements Specification (SRS)

## 1. Introduction
### 1.1 Purpose
The purpose of this Software Requirements Specification (SRS) document is to provide a detailed overview of the software requirements for the "All In One Dictionary" project. It describes the scope of the system, both functional and non-functional requirements, the expected environment, and the interactions between users and the system.

### 1.2 Document Conventions
This document follows the IEEE 830 standard template. The following conventions are used:
- **Bold** text is used to highlight important concepts, fields, or terms.
- `Monospace` text is used for file names, code snippets, and system properties.
- Requirements are prioritized as Essential, Desirable, and Optional.

### 1.3 Intended Audience and Reading Suggestions
This document is intended for:
- **Developers** implementing the frontend application and future backend features.
- **Project Managers** overseeing the product timeline and feature completion.
- **Quality Assurance (QA) Testers** verifying that the final product meets the specified requirements.
- **End Users** seeking to understand the capabilities and limitations of the system.

It is recommended to read the Overall Description first for a high-level overview, followed by the System Features section for specific functional requirements.

### 1.4 Product Scope
The "All In One Dictionary" is a frontend-only offline multilingual dictionary and translator. It is designed to provide rapid translation, search autocomplete, word details, and offline access without requiring active server connections. By utilizing browser capabilities such as the Speech Recognition API and Speech Synthesis API, it offers an enhanced user experience while persisting data via Browser Local Storage.

## 2. Overall Description

### 2.1 Product Perspective
The system is an independent, single-page web application (SPA) built using React 19, TypeScript, Vite, and Tailwind CSS. It currently does not rely on a backend server or external database for basic translation, using an embedded dictionary dataset instead. It interacts with native browser APIs for speech-to-text and text-to-speech functionality and uses the browser's `localStorage` for data persistence.

### 2.2 Product Functions
- **Multilingual Translation:** Direct word, phrase, and fallback sentence translation.
- **Autocomplete Search:** Suggests terms based on prefix and substring matching.
- **Word Detail View:** Displays definition, part of speech, phonetics, examples, synonyms, and antonyms.
- **Language Detection:** Auto-detects the source language based on input.
- **History & Favorites:** Persists up to 50 recent searches and allows users to favorite items.
- **Voice Capabilities:** Speech-to-text input and text-to-speech output.
- **Utilities:** Random word generator, copy-to-clipboard, and source/target language swapping.

### 2.3 User Classes and Characteristics
- **General Users:** Individuals needing quick, offline access to translations, definitions, and pronunciations. They expect a responsive, intuitive, and fast user interface.

### 2.4 Operating Environment
- **Platform:** Modern Web Browsers (Chrome, Edge, Firefox, Safari).
- **Environment:** Client-side execution.
- **APIs Required:** Web Speech API (SpeechRecognition and SpeechSynthesis), LocalStorage.

### 2.5 Design and Implementation Constraints
- **Data Size:** The application's translation capability is bounded by the size of the embedded, static dictionary dataset.
- **Browser Dependency:** Voice input and output rely entirely on the browser's implementation of the Web Speech API. Browsers that do not support these APIs will lack these features.
- **Storage:** LocalStorage is limited (typically ~5MB), restricting the amount of search history and favorites that can be reliably stored.

## 3. External Interface Requirements

### 3.1 User Interfaces
- A responsive, mobile-first Web GUI built with React and Tailwind CSS.
- **Input fields:** Text input for searching/translating words.
- **Controls:** Buttons for swapping languages, toggling voice input, playing audio, clearing history, and marking favorites.
- **Display:** Word detail panels, history list, and autocomplete dropdowns.

### 3.2 Hardware Interfaces
- **Microphone:** Required to capture voice input via the Speech Recognition API.
- **Speakers/Headphones:** Required for the Text-to-Speech audio output.

### 3.3 Software Interfaces
- **Local Storage API:** For persistent storage of `dictionary_favorites` and `dictionary_history`.
- **Web Speech API:** Used for `SpeechRecognition` (voice input) and `SpeechSynthesis` (audio playback).

### 3.4 Communications Interfaces
- In its current state, the app runs entirely offline client-side and requires no network communication once the application assets are loaded.

## 4. System Features

### 4.1 Search and Autocomplete
- **Description:** The system must suggest words and phrases as the user types, matching against English and translated forms.
- **Functional Requirements:**
  - REQ-1.1: System shall display a dropdown list of matching terms based on prefix or substring matching.
  - REQ-1.2: Selecting an autocomplete item shall trigger an immediate translation and detail view.

### 4.2 Translation & Detection
- **Description:** The system must translate words, phrases, and sentences between supported languages.
- **Functional Requirements:**
  - REQ-2.1: The system shall provide exact matches for words and phrases present in the dataset.
  - REQ-2.2: The system shall provide a fallback word-by-word translation for sentences.
  - REQ-2.3: The system shall auto-detect the source language if "Auto Detect" is selected.

### 4.3 Word Detail Panel
- **Description:** Provides comprehensive details for a specific dictionary entry.
- **Functional Requirements:**
  - REQ-3.1: Display the word, phonetic spelling, and part of speech.
  - REQ-3.2: Display definitions, usage examples, synonyms, and antonyms.
  - REQ-3.3: Display mappings to all supported languages for the chosen word.

### 4.4 Voice Input and Text-to-Speech
- **Description:** Allows hands-free operation and pronunciation assistance.
- **Functional Requirements:**
  - REQ-4.1: The user can trigger the microphone to capture spoken words and automatically translate them.
  - REQ-4.2: The user can click an audio icon to hear the pronunciation of the source or translated text.

### 4.5 Favorites and History
- **Description:** Persists user data locally.
- **Functional Requirements:**
  - REQ-5.1: The system shall save the last 50 queries in a History list.
  - REQ-5.2: The system shall allow the user to clear the history.
  - REQ-5.3: The user can toggle a "Favorite" status for any word, saving it to a dedicated list.

## 5. Other Nonfunctional Requirements

### 5.1 Performance Requirements
- **Response Time:** Autocomplete and translation results should render in under 100ms given the local, in-memory nature of the application.
- **Throughput:** Capable of handling rapid consecutive keystrokes without UI blocking.

### 5.2 Safety Requirements
- There are no direct physical safety hazards associated with this software.

### 5.3 Security Requirements
- **Data Privacy:** Because the application is fully client-side, all search history and favorites remain on the user's device and are not transmitted externally.
- No authentication or authorization is required.

### 5.4 Software Quality Attributes
- **Availability:** Near 100% availability since it is an offline PWA/SPA.
- **Maintainability:** Utilizing TypeScript and modular React components ensures code is easily understandable and modifiable.
- **Usability:** The interface is built with Tailwind CSS focusing on an intuitive, modern, accessible, and responsive user experience across devices.

### 5.5 Business Rules
- The application relies on an embedded dataset (`dictionary.ts`). Any updates to the vocabulary require a code deployment, as there is currently no dynamic backend.
